# Übersetzungssystem (i18n)

Drei-Ebenen-Auto-Discovery, ein flat key-value Lookup, sofortiger Sprachwechsel.

## Sprachen

Aktuell: **en** (Default), **de**.

Aktive Sprache wird in `chrome.storage.local` (`wp-lang`) persistiert. Wechsel
per Burger → Einstellungen → Allgemein → EN/DE.

## Translation-Files

Drei Ebenen, alle auto-discovered:

```
src/translations/translations.json                       ← Global (App-Shell)
src/services/<id>/translations/translations.json         ← Pro Service
src/services/<id>/modules/<mid>/translations/...         ← Pro Modul
```

Werden via `import.meta.glob` von `useI18n` eingelesen und in einen flachen
Lookup pro Sprache gemerged. **Konflikte bei gleichen Keys**: das zuletzt
geladene File gewinnt — Keys also distinkt halten oder absichtlich
übersteuern.

## Format

```json
{
  "de": {
    "Hello": "Hallo",
    "Found {count} items": "{count} Treffer gefunden",
    "Save": "Speichern"
  },
  "en": {
    "Save": "Save"
  }
}
```

- **Der Key ist die englische Form.** EN-Übersetzungen sind meist optional —
  wenn der Key fehlt, fällt der Lookup auf den Key selbst zurück.
- DE wird i.d.R. komplett gepflegt.
- Platzhalter im `{name}`-Stil: `t('Hello {name}', { name: 'world' })`.

## Lookup-Reihenfolge

```
merged[currentLang][key] → merged.en[key] → key
```

Fehlt der Key überall, wird der Rohstring (= der Key selbst) zurückgegeben.

## Verwendung

### Sidebar-Code (Vue)

```vue
<script setup>
import { useI18n } from '@/composables/i18n/useI18n.js'
const { t, lang, setLang, supportedLangs } = useI18n()
</script>

<template>
  <p>{{ t('Hello') }}</p>
  <p>{{ t('Found {count} items', { count: 5 }) }}</p>
  <button v-for="code in supportedLangs" :key="code" @click="setLang(code)">
    {{ code.toUpperCase() }}
  </button>
</template>
```

`lang` ist ein Reactive Ref — Änderungen propagieren live, alle `t()`-Aufrufe
re-evaluieren automatisch.

### Page-Kontext (Web-Checker-Checker)

```js
export default function check() {
  const t = window.__t   // injiziert via useCheckRunner.injectHelper
  return {
    errors: [{ message: t('No H1 tag found') }],
    // ...
  }
}
```

`window.__t` und die Translation-Tabelle werden vor jedem Run frisch
injiziert — Sprachwechsel propagiert beim nächsten Recheck.

### Service Worker / `background.js`

Keine Übersetzungsfunktion verfügbar (kein Vue-Kontext). Strings, die der SW
zurückgibt, sollten als englische Keys gesendet werden — der Empfänger im
Sidebar-Code übersetzt dann via `t()`.

## Wo Strings stehen

| Wo | Translation-File |
|---|---|
| Globale UI-Shell (Dashboard, BreadCrumb, AppHeader-Defaults, Settings-Page) | `src/translations/translations.json` |
| Service-Name + Description (`service.json`) | `services/<id>/translations/translations.json` |
| Service-spezifische UI (HomeView-Texte, Toggle-Labels) | `services/<id>/translations/translations.json` |
| Modul-Name + Description (`module.json`) | `services/<id>/modules/<mid>/translations/translations.json` |
| Modul-spezifische UI (Item-Titles, Issue-Messages, Overlay-Buttons) | `services/<id>/modules/<mid>/translations/translations.json` |

## Tipps

- Keys sind **englische Strings** — kein abstrakter ID-Stil wie `module.error_h1_empty`.
  Der Vorteil: Code mit `t('Empty H1')` ist auch ohne Translation lesbar, und
  der Fallback ist sinnvoll.
- Bei Pluralen separate Keys: `'1 item found'` vs `'{count} items found'`.
- Variablen-Namen in Platzhaltern beschreibend: `{count}`, `{tag}`, `{filename}`.
- `name`/`description` aus `service.json` und `module.json` werden auch durch
  `t()` gerendert — dort also den englischen String reinschreiben.
- Klein-Letter-Badges (z.B. `t('Err')` → `'F'` in DE, `'E'` in EN) sind OK,
  aber konsistent benennen.

## Sprache hinzufügen

Aktuell hardcoded auf `en` + `de`. Neue Sprache hinzufügen:

1. In `useI18n.js`: `const SUPPORTED = ['en', 'de', 'fr']`
2. Pro Translation-File: `"fr": { ... }` Block hinzufügen (oder leer lassen)
3. UI: `LangSwitcher`/Settings-Page iteriert über `supportedLangs` — kein
   manueller Eintrag nötig
